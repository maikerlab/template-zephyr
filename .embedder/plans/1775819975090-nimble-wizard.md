### 5. Architecture -- Writing Application Code

Explain the `hal_iface_t` pattern with a walkthrough:

1. **Define the interface** -- `include/hal_iface.h` contains `hal_iface_t`, a struct of function pointers (e.g., `led_toggle`, `btn_register_cb`). To add new hardware capabilities, add a new function pointer here.

2. **Implement for hardware** -- `src/hal/hw_init.c` provides static functions that implement each function pointer using real Zephyr drivers, wires them into a static `hal_iface_t`, and returns it from `hw_init()`. Add the real driver calls here.

3. **Use in application** -- `src/main.c` calls `hw_init()` to get the interface, then uses it through the function pointers. Application code never calls drivers directly.

4. **Register new source files** -- add any new `.c` files to `target_sources(app PRIVATE ...)` in the root `CMakeLists.txt`.

Include a brief code snippet showing how to extend the interface (add a function pointer, implement it in `hw_init.c`, call it in `main.c`).

### 6. Writing Tests

Explain the mock-based testing approach:

1. **Mock implementations** -- `tests/unit/mock_hal.h` provides stub implementations for every function pointer in `hal_iface_t` and a `mock_hal_create()` helper to construct a mock interface. When adding a new function pointer to `hal_iface_t`, add a corresponding mock here.

2. **Create a test suite** -- add a new `.c` file in `tests/unit/src/`. Include `<zephyr/ztest.h>` and `"mock_hal.h"`. Register a suite with `ZTEST_SUITE()` and define test cases with `ZTEST()`.

3. **Register the test source** -- add the new file to `target_sources(app PRIVATE ...)` in `tests/unit/CMakeLists.txt`.

4. **Key rule** -- tests never link `hw_init.c`. They construct their own `hal_iface_t` with mock function pointers, so no hardware dependencies are pulled in.

Include a minimal example of a test file.

### 7. Running Tests

Two methods:

**A. On host with native_sim (fast iteration, no hardware needed)**

```bash
# Run all unit tests
west twister -T tests/unit -p native_sim

# Verbose output
west twister -T tests/unit -p native_sim -v
```

**B. On-device on the nRF7002 DK**

```bash
# Build tests for the board
west build -b nrf7002dk/nrf5340/cpuapp -p always tests/unit

# Flash and observe output on serial
west flash
```

Mention that test results appear on the serial console (115200 baud) and include a sample output snippet showing what a passing/failing run looks like.

### 8. Code Style (brief)

- Zephyr coding guidelines
- Tabs, K&R braces, snake_case, `_t` suffix for typedefs
- Zephyr logging (`LOG_INF`, `LOG_ERR`) instead of `printk`
- HAL functions return `int` (0 success, negative errno)
